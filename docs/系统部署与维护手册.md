## 1. 概述

### 1.1 编写目的

本手册记录农业资源数字化管理平台(以下简称"本系统")的服务器资源信息、运行环境、部署步骤与日常运维操作，供后续运维/交接人员参考。

### 1.2 部署架构

本系统采用 Docker Compose 容器化部署，单台云主机内运行 3 个容器，互不依赖宿主机安装 JDK/MySQL/Nginx：

| 容器 | 镜像 | 作用 | 对外端口 |
|---|---|---|---|
| nyzy-mysql | mysql:5.7 (platform linux/amd64) | 数据存储 | 13306→3306(仅宿主机内部) |
| nyzy-backend | 自建(JDK8 + Spring Boot, 见 backend/Dockerfile) | 业务API | 8080(仅宿主机内部) |
| nyzy-frontend | 自建(nginx:1.25-alpine + Vue3静态资源, 见 frontend/Dockerfile) | 前端页面 + API反向代理 | 8000→80(对公网开放) |

请求路径：浏览器 → 云主机:8000(frontend容器nginx) → 静态资源直接返回 / `/api/` 前缀反代到 backend:8080 → backend 容器 → mysql:3306。

## 3. 运行环境安装

本系统不依赖宿主机单独安装 JDK / MySQL / Nginx / Redis，仅需安装 **Docker + Docker Compose**。

### 3.1 安装 Docker(Amazon Linux 2023)

```bash
sudo dnf update -y
sudo dnf install -y docker
sudo systemctl enable --now docker
sudo usermod -aG docker ec2-user   # 免sudo执行docker命令，重新登录后生效
```

### 3.2 安装 Docker Compose 插件

```bash
sudo dnf install -y docker-compose-plugin
docker compose version   # 验证安装成功
```

> 若云主机镜像不含上述 yum 源，可改用官方二进制安装方式，详见 Docker 官方文档，不在此重复。

## 4. 应用部署

### 4.1 获取代码

```bash
cd /opt
sudo git clone https://github.com/selinawxyi/nyzy.git nyzy
cd nyzy
```

> 仓库为公开仓库，可匿名 clone，无需配置部署密钥。

### 4.2 配置说明

核心配置在仓库根目录 `docker-compose.yml`，无需额外创建 `.env` 文件，关键环境变量：

| 变量 | 默认值 | 说明 |
|---|---|---|
| MYSQL_ROOT_PASSWORD | nyzy123456 | 数据库root密码，如需改密需同步改 backend 的 SPRING_DATASOURCE_PASSWORD |
| MYSQL_DATABASE | nyzy | 数据库名 |
| SPRING_DATASOURCE_URL | jdbc:mysql://mysql:3306/nyzy?... | backend连接数据库地址(容器间用服务名mysql访问，不是localhost) |
| NYZY_UPLOAD_DIR | /app/uploads | 后端文件上传存储路径(容器内)，对应 uploads-data 数据卷 |

数据持久化用两个 Docker named volume：`mysql-data`(数据库文件)、`uploads-data`(用户上传的附件/图片)，删除容器不会丢数据，删除 volume 才会丢。

### 4.3 首次启动

```bash
cd /opt/nyzy
sudo docker compose up -d --build
```

首次启动会：① 构建 backend 镜像(maven 编译，约2-3分钟)；② 构建 frontend 镜像(npm build)；③ 拉取 mysql:5.7 镜像并执行 `db/init/` 目录下全部 `*.sql` 初始化脚本(建表+种子数据)。

### 4.4 启动 / 停止 / 重启命令

```bash
# 启动(已构建过镜像，不重新构建)
docker compose up -d

# 停止(容器停止，数据卷保留)
docker compose stop

# 重启单个服务
docker compose restart backend

# 完全停止并删除容器(不加 -v 不会删数据卷)
docker compose down

# 查看容器状态
docker compose ps
```

### 4.5 更新部署(代码变更后)

本项目目前**没有 CI/CD 自动化**，更新部署需手动执行：

```bash
ssh -i ~/.ssh/nyzy-deploy.pem ec2-user@<EC2公网IP>
cd /opt/nyzy
sudo git pull
sudo docker compose up -d --build
```

`--build` 会重新编译有改动的服务镜像，未改动的服务(如未变更代码的 mysql)不受影响。

#### 4.5.1 数据库结构变更(新增 db/init 脚本)需手动迁移

`db/init/` 目录下的 SQL 脚本只会在 `mysql-data` 数据卷**首次初始化**(即数据卷不存在)时自动执行一次。生产库的数据卷早已存在，之后新增的 `db/init/NN_xxx.sql` **不会自动重跑**，必须手动对生产库执行一遍，否则 backend 启动正常但访问相关功能会报 `Unknown column` 500 错误：

```bash
docker exec -i nyzy-mysql mysql -uroot -pnyzy123456 nyzy < db/init/12_xxx.sql
```

每次给现有表加字段类的迁移脚本，部署后都要走这一步，不要遗漏。

## 5. 日常维护操作

### 5.1 查看日志

```bash
docker logs -f nyzy-backend     # 后端应用日志(含SQL异常/接口报错)
docker logs -f nyzy-frontend    # nginx access/error日志
docker logs -f nyzy-mysql       # 数据库日志
```

### 5.2 数据库备份

```bash
docker exec nyzy-mysql mysqldump -uroot -pnyzy123456 nyzy > backup_$(date +%Y%m%d).sql
```

建议定期(如每日)执行并将备份文件传出云主机保存，当前未配置自动备份/定时快照。

### 5.3 数据库恢复

```bash
docker exec -i nyzy-mysql mysql -uroot -pnyzy123456 nyzy < backup_20260101.sql
```

### 5.4 进入容器调试

```bash
docker exec -it nyzy-mysql mysql -uroot -pnyzy123456 nyzy   # 进MySQL命令行
docker exec -it nyzy-backend sh                              # 进backend容器(JRE-alpine无bash，用sh)
```

### 5.5 健康检查

`GET /api/system/health`(经前端反代即 `http://<公网IP>:8000/api/system/health`)，返回 `code:0` 即应用正常。MySQL 容器自带 `mysqladmin ping` healthcheck，可用 `docker compose ps` 查看健康状态列。

### 5.6 常见问题

| 现象 | 原因 | 处理 |
|---|---|---|
| `docker compose up` 时 mysql 容器报架构不兼容 | mysql:5.7 官方镜像无 arm64 版本(如在 Apple Silicon 本机/ARM云主机部署) | compose 文件已固定 `platform: linux/amd64`，云主机用 x86_64 机型(如 t3 系列)即可，无需改动 |
| backend 编译时 OOM 被杀 | 云主机内存太小(如1GB实例)，maven编译吃内存 | 加 swap 文件(本部署已加2GB swap)，或换内存更大机型 |
| 新功能访问报 500 `Unknown column` | 新增的 db/init 脚本未手动跑到生产库 | 见 4.5.1，手动执行对应SQL文件 |
| 重启实例后前端访问不了 | 未绑定 Elastic IP，重启后公网IP变化 | 重新查询当前公网IP；如需长期稳定访问，建议分配 Elastic IP |
| 中文参数(如删除原因)接口报错/静默失败 | Tomcat 拒绝未编码的多字节字符 GET 参数；docker exec 进 mysql 客户端时字符集不匹配 | curl 测试需 `--data-urlencode`；前端 axios 自动编码无此问题；docker exec 内按中文条件 WHERE 删数据建议改用主键 id |

## 6. 后续可考虑的生产化改进

当前为"先跑起来"的临时方案，非生产级别，后续如需长期/正式对外提供服务，可考虑：

- 分配 Elastic IP，避免重启换IP
- 套 ACM + ALB(或 Nginx+Certbot)做 HTTPS，目前是纯 HTTP
- 用 RDS 替换容器内自建 MySQL，获得自动备份/高可用
- 接入 CI/CD(如 GitHub Actions)，替代手动 SSH + git pull 部署
- 修改所有默认密码(见 2.3 节)
- 定时快照/异地备份策略
